# PLAN — SampleOrderSystem 구현 계획 (초안)

이 문서는 `docs/PRD.md`의 요구사항을 실제로 구현하기 위한 Phase별 작업 계획 초안입니다. 각 Phase는 이전 Phase가 끝나야 다음으로 넘어갈 수 있도록 순서를 잡았고, Phase마다 **직접 눈으로 확인/검증할 수 있는 방법**을 함께 적었습니다. 진행하면서 범위가 바뀌면 이 문서를 갱신하세요.

> 이 계획은 초안이며, 각 Phase 착수 전 세부 설계(특히 "설계 결정 필요" 표시된 부분)를 먼저 확정하는 것을 권장합니다.

> **개발 방법론 — TDD 적용**: Phase 0 이후의 모든 프로덕션 코드(신규 기능, 버그 수정, 리팩터링 포함)는 `test-driven-development` 스킬(`.claude/skills/test-driven-development/SKILL.md`)을 적용하여 개발합니다. 즉, 각 Phase의 작업 항목마다 (1) 실패하는 pytest 테스트를 먼저 작성 → (2) 실패하는 것을 직접 확인 → (3) 테스트를 통과시키는 최소한의 코드 작성 → (4) 리팩터링 순서로 진행하고, 아래 각 Phase의 "검증 방법" 체크리스트는 이 TDD 사이클의 테스트 케이스 목록으로도 활용합니다.

## Phase 0 — 프로젝트 뼈대 & 데이터 스키마 확정

**목표**: 앱 코드가 들어갈 구조를 만들고, `poc_json` 위에서 시료/주문 데이터가 어떤 형태(JSON 스키마)로 저장될지 확정한다.

**작업 내용**
- 애플리케이션 패키지(예: `sample_order_system/`) 생성: `model.py`, `view.py`, `controller.py`, `main.py` 등 `mvc` 라이브러리를 상속하는 구조.
- 데이터 파일 경로 정의: 예) `data/samples.json`, `data/orders.json`.
- Sample/Order 레코드 필드 확정 (PRD 5장 기준: 시료 ID/이름/평균 생산시간/수율/재고, 주문번호/시료ID/고객명/수량/상태).
- 주문번호 채번 규칙 확정 (예: `ORD-YYYYMMDD-NNNN` 또는 단순 증가 정수 — PRD는 형식을 강제하지 않으므로 자유 결정).
- pytest 설치 및 `tests/` 디렉토리, `pytest.ini`/`pyproject.toml` 최소 설정.
- **설계 결정 필요**: 생산 진행 시간을 어떻게 표현할지 — (a) 실제 wall-clock 시간 기반(주문 승인 시각 + 누적 소요시간으로 완료 예정 시각 계산), (b) 콘솔에서 "생산 진행" 같은 수동 tick 명령으로 진행시키는 방식. PRD 4.6은 "진행율/남은 시간/예상 완료시간"을 요구하므로 둘 중 하나를 정하고 PRD/PLAN에 기록.

**검증 방법 (직접 테스트)**
- [ ] 저장소를 새로 클론/체크아웃한 상태에서 `.venv/Scripts/python.exe -m pytest`가 (테스트가 없더라도) 에러 없이 수집·실행되는지 확인.
- [ ] `python -c "import sample_order_system"` 등으로 패키지가 정상 import 되는지 확인.
- [ ] `data/samples.json`, `data/orders.json`이 처음엔 없다가, `poc_json`의 `create_item` 등을 한 번 호출하면 자동 생성되는지 확인 (`poc_json.json_crud._load_records`가 파일 없을 때 빈 리스트 반환하는 동작 활용).

## Phase 1 — 시료(Sample) 도메인

**목표**: PRD 4.2 "시료 관리" 기능을 구현한다.

**작업 내용**
- 시료 등록: 시료 ID, 이름, 평균 생산시간, 수율 입력 → `poc_json.create_item`으로 저장(초기 재고는 0 또는 입력받은 값).
- 시료 조회: 전체 목록 + 현재 재고를 표 형태로 출력.
- 시료 검색: 이름 등으로 필터링해 동일한 표 형식으로 출력.
- 입력 검증: 중복 ID 등록 방지(`create_item`이 이미 처리), 수율 0~1 범위 검증, 평균 생산시간 양수 검증.

**검증 방법 (직접 테스트)**
- [ ] 앱 실행 후 `[1] 시료 관리 → 시료 등록`에서 새 시료(예: `S-001`, "실리콘 웨이퍼-8인치", 평균생산시간 0.5, 수율 0.92) 등록 → 정상 등록 메시지 확인.
- [ ] 같은 ID(`S-001`)로 다시 등록 시도 → 에러 메시지로 거부되는지 확인(중복 방지).
- [ ] `시료 조회` 실행 → 방금 등록한 시료가 ID/이름/평균 생산시간/수율/재고 5개 항목과 함께 나오는지 확인.
- [ ] `시료 검색`에서 이름 일부(예: "실리콘")로 검색 → 해당 시료만 나오는지 확인. 존재하지 않는 이름 검색 시 빈 결과 처리 확인.
- [ ] `data/samples.json` 파일을 직접 열어 등록한 내용이 그대로 저장되어 있는지 확인, 또는 `python -m poc_json.admin_tool data/samples.json`로 `list` 명령 실행해 확인.
- [ ] pytest: 시료 등록/중복 등록 실패/조회/검색에 대한 단위 테스트 작성 및 통과 확인.

## Phase 2 — 주문 예약 (Reserve)

**목표**: PRD 4.3 "시료 주문"의 예약 기능을 구현한다.

**작업 내용**
- 주문 생성 시 입력: 시료 ID, 고객명, 주문 수량.
- 등록되지 않은 시료 ID로 주문 시 에러 처리.
- 생성된 주문은 상태 `RESERVED`로 `orders.json`에 저장.

**검증 방법 (직접 테스트)**
- [ ] `[2] 시료 주문`에서 Phase 1에서 등록한 시료 ID로 주문(예: 고객 "삼성전자 파운드리", 수량 200) 생성 → 주문번호와 상태 `RESERVED`가 표시되는지 확인.
- [ ] 존재하지 않는 시료 ID(예: `S-999`)로 주문 시도 → 에러 메시지로 거부되는지 확인.
- [ ] `data/orders.json` 확인 또는 `poc_json.admin_tool`로 방금 생성한 주문이 `RESERVED` 상태로 저장되어 있는지 확인.
- [ ] pytest: 정상 예약, 존재하지 않는 시료 예약 실패 케이스 테스트.

## Phase 3 — 주문 승인/거절

**목표**: PRD 4.4를 구현한다. 이 Phase가 재고 판정과 생산 큐 등록의 핵심 분기점이다.

**작업 내용**
- `RESERVED` 주문 목록을 번호(1,2,3...)와 함께 표시.
- 번호 선택 시 해당 시료 정보, 부족분(`주문수량 - 재고`), 재고량을 보여주고 부족한 경우 예상 소요시간(생산 큐에 이미 대기 중인 작업들의 누적 시간 + 이번 건 소요시간)을 계산해서 보여준다.
- 승인 선택 시:
  - 재고 충분 → 즉시 `CONFIRMED`.
  - 재고 부족 → 생산 큐에 등록, 주문 상태 `PRODUCING`.
- 거절 선택 시 → 즉시 `REJECTED`.

**검증 방법 (직접 테스트)**
- [ ] **재고 충분 케이스**: 시료 재고를 주문 수량 이상으로 맞춘 뒤(Phase 1에서 재고 보정 기능이 없다면 이 Phase 전에 임시로 데이터 파일을 직접 편집해도 됨) 주문 승인 → 상태가 즉시 `CONFIRMED`로 바뀌는지, 재고가 아직 차감되지 않았는지(차감은 출고 시점) 확인.
- [ ] **재고 부족 케이스**: 재고보다 큰 수량으로 주문 후 승인 → 부족분/예상 소요시간이 화면에 표시되는지, 승인 후 상태가 `PRODUCING`으로 바뀌는지, 생산 큐에 항목이 추가되는지(Phase 4 화면 또는 데이터 파일로 확인) 체크.
- [ ] **거절 케이스**: `RESERVED` 주문을 거절 → 상태가 `REJECTED`로 바뀌고 이후 승인/거절 목록에 다시 나타나지 않는지 확인.
- [ ] 이미 처리된 주문(승인/거절 완료)이 "접수된 주문 목록"에서 더 이상 보이지 않는지 확인.
- [ ] pytest: 재고 충분/부족/거절 3가지 분기, 존재하지 않는 번호 선택 시 에러 처리 테스트.

## Phase 4 — 생산 라인 (FIFO)

**목표**: PRD 4.6을 구현한다. Phase 3에서 등록된 생산 큐 항목을 실제로 처리하는 로직.

**작업 내용**
- 생산 큐 자료구조(FIFO) — 큐에는 주문번호, 시료ID, 부족분, 실생산량(`ceil(부족분/수율)`), 총 생산시간(`평균생산시간 * 실생산량`), 시작 시각 등을 저장.
- 생산 현황 화면: 현재 처리 중 항목의 시료 이름/주문량/재고/부족량/실생산량/진행율/남은 시간.
- 대기 목록 화면: FIFO 순서, 주문번호, 시료 이름, 주문량, 부족분, 실생산량, 예상 완료시간.
- 생산 완료 처리: Phase 0에서 정한 시간 진행 방식에 따라 완료된 건에 대해 재고를 실생산량만큼 증가시키고 주문 상태를 `PRODUCING → CONFIRMED`로 전환. 완료된 항목은 큐에서 제거하고 다음 대기 항목이 "처리 중"으로 승격.
- "뒤로" 메뉴 포함.

**검증 방법 (직접 테스트)**
- [ ] Phase 3에서 재고 부족으로 `PRODUCING` 상태가 된 주문이 `[5] 생산 라인 조회`의 "현재 처리 중"에 나오는지 확인.
- [ ] 부족분/실생산량 계산이 `ceil(부족분/수율)` 공식과 일치하는지 손으로 계산해 검증 (예: 부족분 170, 수율 0.92 → 실생산량 = ceil(170/0.92) = 185).
- [ ] 여러 건을 연속 승인해 생산 큐에 2건 이상 쌓았을 때, 대기 목록이 접수(승인)된 순서(FIFO)대로 나오는지 확인.
- [ ] 시간 진행 방식(수동 tick 또는 wall-clock)에 맞춰 생산을 완료시킨 뒤: 해당 주문 상태가 `CONFIRMED`로 바뀌는지, 시료 재고가 실생산량만큼 늘어났는지(`시료 조회`로 확인), 다음 대기 항목이 처리 중으로 넘어가는지 확인.
- [ ] pytest: 실생산량/총생산시간 계산 공식에 대한 단위 테스트(경계값 포함 — 부족분이 수율로 나누어 떨어지지 않는 경우), FIFO 순서 테스트, 생산 완료 후 상태 전이 및 재고 반영 테스트.

## Phase 5 — 출고 처리

**목표**: PRD 4.7을 구현한다.

**작업 내용**
- `CONFIRMED` 상태 주문만 번호/주문번호/고객/시료이름/수량과 함께 목록 표시.
- 번호 선택 시 해당 주문 상태를 `RELEASE`로 전환, 처리 결과(처리일시 등) 표시.
- 출고 수량만큼 해당 시료 재고 차감.
- "뒤로" 메뉴 포함.

**검증 방법 (직접 테스트)**
- [ ] Phase 3/4에서 `CONFIRMED`가 된 주문이 `[6] 출고 처리` 목록에 나오는지 확인.
- [ ] 출고 실행 전후로 `시료 조회`의 재고 수량이 정확히 주문 수량만큼 줄어드는지 확인.
- [ ] 출고 실행 후 해당 주문이 상태 `RELEASE`로 바뀌고, 다시 출고 목록에는 나타나지 않는지 확인.
- [ ] 재고가 음수가 되지 않는지(출고 가능한 만큼만 CONFIRMED로 왔어야 하므로) 확인 — 만약 음수가 나온다면 Phase 3 로직 재검토.
- [ ] pytest: 출고 처리 시 상태 전이 및 재고 차감 정확성 테스트, `CONFIRMED`가 아닌 주문은 출고 목록에 나오지 않는지 테스트.

## Phase 6 — 모니터링

**목표**: PRD 4.5를 구현한다.

**작업 내용**
- 주문량 확인: `RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE` 각각의 건수 집계 (`REJECTED` 제외).
- 재고량 확인: 시료별 재고 내림차순 정렬, 여유/부족/고갈 상태 계산 로직 정의.
  - **설계 결정 필요**: "주문 대비"의 기준이 되는 미출고 수요를 어떻게 집계할지 (예: 해당 시료에 대해 아직 `RELEASE`되지 않은 주문 수량 합) 확정.
- "뒤로" 메뉴 포함.

**검증 방법 (직접 테스트)**
- [ ] 지금까지의 테스트로 만들어진 데이터 상태에서 `[4] 모니터링 → 주문량 확인` 실행 → 각 상태별 건수를 손으로 센 값과 비교해 일치하는지 확인. `REJECTED` 건수는 집계에 포함되지 않는지 확인.
- [ ] `재고량 확인` 실행 → 재고 많은 시료부터 내림차순으로 나오는지 확인.
- [ ] 재고가 0인 시료가 "고갈"로, 미출고 수요보다 적은 시료가 "부족"으로, 충분한 시료가 "여유"로 표시되는지 각각 하나씩 케이스를 만들어 확인.
- [ ] pytest: 상태별 집계 함수, 여유/부족/고갈 분류 함수에 대한 경계값 테스트(재고=0, 재고=수요와 동일, 재고>수요).

**목표**: `dummy_data_generator`를 이용해 테스트/데모용 초기 데이터를 만든다.

## 부록 — 전체 End-to-End 수동 테스트 시나리오 (요약)

앱이 어느 정도 완성된 뒤, 아래 순서대로 한 번에 실행해보면 전체 기능이 유기적으로 맞물려 동작하는지 빠르게 확인할 수 있습니다.

1. 시료 2종 등록 (하나는 재고를 넉넉히, 하나는 재고를 적게 설정).
2. 재고가 넉넉한 시료로 소량 주문 → 승인 → 즉시 `CONFIRMED`인지 확인.
3. 재고가 적은 시료로 재고 초과 수량 주문 → 승인 → `PRODUCING`, 생산 큐에 등록되는지 확인.
4. 생산 라인 조회에서 처리 중/대기 정보가 올바른지 확인.
5. 생산 완료 후 해당 주문이 `CONFIRMED`로 바뀌고 재고가 늘어났는지 확인.
6. 두 `CONFIRMED` 주문을 모두 출고 처리 → 재고 차감과 상태 `RELEASE` 확인.
7. 모니터링에서 `RELEASE` 2건, 나머지 상태 0건으로 집계되는지, 재고 상태(여유/부족/고갈)가 최종 재고량과 일치하는지 확인.
8. 중간에 주문 하나를 거절해보고 `REJECTED`가 모니터링 집계에서 제외되는지 확인.
